# “AI思政”辅助APP教师端页面设计方案

## 一、教师端定位

教师端主要服务辅导员、班主任、任课教师、实习指导教师等角色，用于创建和管理面向学生的打卡任务。

教师端不是固定发布“查寝、上课、实习”三类打卡，而是支持教师基于班级或分组创建不同类型的打卡任务，并通过灵活配置规则满足不同管理场景。

教师端核心定位：

> 教师端是面向班级和学生群体的打卡任务配置、发布、统计和异常处理工具。教师可以选择预设打卡类型，也可以自定义打卡任务，通过配置时间、地点、身份校验、提交内容和审核规则，实现多场景学生管理。

---

# 二、教师端底部导航设计

教师端建议采用小程序，底部设置四个一级导航：

1. 首页
2. 任务
3. 班级
4. 我的

也可以把“消息”合并到首页和我的中，不单独作为一级导航，避免教师端过重。

---

## 一级导航说明

| 导航 | 作用                |
| -- | ----------------- |
| 首页 | 今日任务概览、异常提醒、待处理事项 |
| 任务 | 创建任务、管理任务、查看任务数据  |
| 班级 | 管理班级、查看学生、班级打卡情况  |
| 我的 | 教师信息、权限、消息、设置、帮助  |

---

# 三、教师端核心业务流程

教师端主流程建议设计为：

1. 选择班级 / 分组
2. 选择或创建打卡类型
3. 配置打卡规则
4. 发布打卡任务
5. 学生完成打卡
6. 教师查看数据
7. 教师处理异常 / 审核申诉
8. 系统形成统计记录

---

# 四、教师端页面一：首页

## 页面名称

首页 / 工作台

## 页面目标

让教师快速了解今天需要关注的班级任务、学生完成情况和异常事项。

---

## 页面结构

### 1. 顶部教师信息区

展示：

* 教师姓名
* 教师角色
* 所属学院 / 系部
* 管理班级数量
* 今日日期

示例：

**张老师**
辅导员 / 2024级软件技术1班
今日：2026年6月24日

---

### 2. 今日工作概览

以数据卡片形式展示：

* 今日发布任务数
* 待完成学生数
* 异常打卡数
* 待审核申诉数

示例：

| 指标   | 数量 |
| ---- | -- |
| 今日任务 | 4  |
| 待完成  | 26 |
| 异常记录 | 3  |
| 待审核  | 2  |

---

### 3. 待处理事项

展示需要教师立即处理的事项。

例如：

* 早操签到有 3 名学生定位异常
* 晚间查寝有 2 名学生未打卡
* 实习日报有 5 名学生未提交
* 有 2 条异常申诉待审核

每条待处理事项可以点击进入对应详情页。

---

### 4. 快捷操作区

建议放置教师常用操作：

* 创建打卡任务
* 查看今日任务
* 异常处理
* 班级数据
* 学生查询

---

### 5. 最近任务动态

展示教师最近发布的任务及完成情况。

示例：

**晚间宿舍安全打卡**
班级：2024级软件技术1班
完成率：86%
异常：2人
按钮：查看详情

**人工智能导论课堂签到**
班级：人工智能导论A班
完成率：94%
异常：1人
按钮：查看详情

# 六、教师端页面三：创建打卡任务页

这是教师端最核心的页面。

## 页面名称

创建打卡任务

## 页面目标

教师通过选择班级、打卡类型和规则配置，快速创建不同场景的打卡任务。

---

## 创建流程建议

创建任务可以设计为分步骤表单，降低教师使用难度。

建议分为五步：

1. 基础信息
2. 选择对象
3. 选择打卡类型
4. 配置打卡规则
5. 确认发布

---

## 第一步：基础信息

字段包括：

* 任务名称
* 任务说明
* 发布人
* 是否保存为模板
* 是否发送通知

示例：

任务名称：晚间宿舍安全打卡
任务说明：请同学们在规定时间内于宿舍区域完成打卡。

---

## 第二步：选择对象

因为系统以班级管理为核心，所以这里非常重要。

教师可以选择：

* 行政班
* 课程班
* 实习小组
* 活动分组
* 自定义学生范围

支持多选班级，也支持排除个别学生。

字段包括：

* 选择班级
* 选择学生范围
* 是否包含请假学生
* 是否仅对部分学生发布

示例：

发布对象：2024级软件技术1班
参与人数：48人
排除学生：2人，已请假

---

## 第三步：选择打卡类型

教师可以选择系统预设类型，也可以选择自定义类型。

### 预设打卡类型

| 类型   | 适用场景          |
| ---- | ------------- |
| 查寝类  | 晚间归寝、安全确认     |
| 课堂类  | 上课签到、课堂考勤     |
| 实习类  | 校外实习、岗位实践     |
| 活动类  | 班会、讲座、志愿活动    |
| 会议类  | 学生干部会议、主题教育会议 |
| 早操类  | 晨练、早操、晨读      |
| 返校类  | 假期返校、离校返校确认   |
| 自定义类 | 教师自行配置        |

---

## 第四步：配置打卡规则

这是系统可扩展性的核心。

教师端应支持按模块配置规则。

---

## 1. 时间规则

支持：

* 单次打卡
* 每日打卡
* 每周重复
* 多时间段打卡
* 上班 / 下班两次打卡
* 自定义周期
* 允许迟到
* 允许补卡
* 截止后不可提交
* 截止后提交需审核

示例配置：

打卡时间：21:30—22:30
补卡规则：截止后可提交说明，但需教师审核

---

## 2. 地点规则

支持：

* 不限制地点
* 限制学校范围
* 限制教学楼范围
* 限制宿舍楼范围
* 限制实习单位范围
* 自定义地点
* 多地点可选
* 允许外勤说明

字段包括：

* 打卡地点
* 有效范围
* 定位偏差容许范围
* 是否允许定位异常提交说明

示例配置：

地点：3号宿舍楼
有效范围：300米内
异常处理：允许提交说明，教师审核

---

## 3. 身份校验规则

支持：

* 无需校验
* 人脸识别
* 动态码
* 二维码
* 教师现场确认
* 定位 + 动态码
* 定位 + 人脸识别
* 动态码 + 人脸识别

示例配置：

身份校验方式：定位 + 人脸识别

---

## 4. 提交内容规则

支持教师自定义学生需要提交的内容。

可选组件包括：

* 文字说明
* 图片上传
* 实习日志
* 安全状态
* 表单问卷
* 今日总结
* 风险上报
* 附件上传

教师可以设置字段是否必填。

示例：

提交内容：

* 今日工作内容，必填
* 今日收获，必填
* 是否存在安全风险，必填
* 图片上传，选填

---

## 5. 审核规则

支持：

* 自动通过
* 异常才审核
* 所有记录需审核
* 补卡需审核
* 定位异常需审核
* 人脸识别失败需审核
* 学生申诉需审核

示例：

审核规则：正常打卡自动通过，异常打卡进入待审核。

---

## 6. 提醒规则

支持：

* 开始前提醒
* 截止前提醒
* 未完成提醒
* 异常提醒
* 审核结果通知

示例：

开始前10分钟提醒学生
截止前5分钟提醒未完成学生
任务结束后提醒教师查看异常数据

---

## 第五步：确认发布

发布前展示完整配置摘要：

* 任务名称
* 发布班级
* 参与人数
* 打卡类型
* 打卡时间
* 地点规则
* 校验方式
* 提交内容
* 审核规则
* 通知方式

按钮：

* 保存草稿
* 保存为模板
* 立即发布
* 定时发布

---

* 待审核数

# 十二、教师端页面九：数据统计页

这个页面可以作为任务详情或班级详情里的二级页面，不一定放在底部导航。

## 页面名称

数据统计

## 页面目标

帮助教师从整体上掌握学生出勤、打卡、异常和成长情况。

---

## 统计维度

### 按班级统计

* 班级任务完成率
* 班级异常率
* 班级未完成名单
* 班级连续完成情况

### 按学生统计

* 学生个人任务完成率
* 异常次数
* 迟到次数
* 申诉次数
* 实习日志完成情况

### 按任务类型统计

* 查寝类完成率
* 课堂类完成率
* 实习类完成率
* 活动类完成率
* 自定义类型完成率

### 按时间统计

* 日统计
* 周统计
* 月统计
* 学期统计

---

## 数据用途

统计页不仅用于管理，也可以服务思政教育和学生成长引导。

例如：

* 对长期未完成学生进行重点关注
* 对连续完成学生进行正向激励
* 对异常高发场景优化管理方式
* 对实习风险学生及时跟进

# 十六、教师端设计重点

## 1. 以班级为中心

教师端所有任务都应围绕班级或分组创建，不能只围绕单个打卡类型。这样才能支持行政班、课程班、实习小组、活动分组等多种管理场景。

---

## 2. 以规则配置实现扩展

教师端不应该为每一种新打卡场景单独开发新功能，而是通过规则配置实现。

可配置规则包括：

* 时间规则
* 地点规则
* 身份校验规则
* 提交内容规则
* 审核规则
* 提醒规则

---

## 3. 预设模板降低使用门槛

虽然系统支持自定义，但教师不一定愿意每次从零配置。因此需要提供常用模板，例如查寝、课堂、实习、活动签到等。

---

## 4. 异常处理要集中

教师端最重要的管理动作不是“看数据”，而是“处理异常”。因此首页、任务详情和异常处理页都应突出待处理事项。

---

## 5. 数据统计服务思政教育

教师端统计数据不仅是考勤数据，也可以转化为思政管理依据，例如诚信教育、安全教育、学风建设、责任意识培养等。

---

# 十七、教师端与学生端的对应关系

| 教师端操作  | 学生端表现                |
| ------ | -------------------- |
| 创建打卡任务 | 学生首页出现待完成任务          |
| 选择班级   | 对应班级学生收到任务           |
| 配置时间规则 | 学生端显示打卡时间和倒计时        |
| 配置地点规则 | 学生端加载定位组件            |
| 配置身份校验 | 学生端加载人脸 / 动态码 / 二维码  |
| 配置提交内容 | 学生端显示日志 / 图片 / 表单    |
| 设置审核规则 | 学生端显示待审核 / 已通过 / 已驳回 |
| 处理异常申诉 | 学生端收到申诉结果通知          |
| 发送提醒   | 学生端收到打卡提醒消息          |

# 十八、最终建议

教师端建议定位为：

> 面向教师的班级打卡任务配置与管理小程序。教师可以基于班级或分组发布不同类型的打卡任务，通过预设模板或自定义规则配置时间、地点、身份校验、提交内容和审核方式，并实时查看学生完成情况、处理异常申诉、分析班级数据，从而提升学生管理效率，支撑安全管理、学风建设、诚信教育和实习过程管理。

教师端第一阶段重点做四件事：

1. **创建任务**
2. **管理班级**
3. **查看完成情况**
4. **处理异常申诉**

其他统计分析、模板共享、风险预警等功能可以作为第二阶段增强能力。
