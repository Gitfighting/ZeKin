# “AI思政”辅助APP管理员端 Web 页面设计方案

## 一、管理员端定位

管理员端采用 Web 后台形式，主要面向学校学工处、学院管理员、系统管理员、辅导员负责人、教务管理人员等角色。

管理员端不直接承担具体学生打卡操作，也不主要承担单个班级任务发布，而是负责平台整体管理，包括组织架构、人员权限、班级分组、打卡类型、规则模板、数据统计、异常监管、系统配置等。

管理员端核心定位：

> 管理员端是“AI思政”辅助APP的后台管理与数据监管平台，负责维护学校、学院、班级、教师、学生等基础数据，配置打卡类型和规则模板，管理教师权限，监管全校或学院范围内的打卡任务运行情况，并通过数据分析支撑学生安全管理、学风建设、诚信教育和实习过程管理。

---

# 二、管理员端与学生端、教师端的关系

## 1. 学生端

学生端主要负责：

* 接收班级打卡任务
* 完成定位、人脸、动态码、日志等打卡要求
* 查看个人任务、消息、记录和成长档案
* 提交异常说明或申诉

## 2. 教师端

教师端主要负责：

* 基于班级或分组创建打卡任务
* 选择打卡类型
* 配置任务规则
* 查看学生完成情况
* 处理异常和申诉
* 管理班级数据

## 3. 管理员端

管理员端主要负责：

* 维护组织架构
* 管理教师、学生和班级数据
* 管理打卡类型
* 管理规则模板
* 管理权限角色
* 监管全局任务数据
* 处理平台级异常
* 配置系统参数
* 查看数据分析报表

可以理解为：

| 端口   | 核心职责           |
| ---- | -------------- |
| 学生端  | 完成任务           |
| 教师端  | 创建任务、处理异常      |
| 管理员端 | 配置规则、管理权限、监管数据 |

---

# 四、页面一：工作台

## 页面名称

管理员工作台

## 页面目标

让管理员快速了解平台整体运行状态，包括今日任务数量、学生完成情况、异常情况、教师使用情况和风险提醒。

---

## 页面布局

### 1. 顶部数据概览区

以数据卡片展示核心指标：

* 今日打卡任务数
* 今日参与学生数
* 今日完成率
* 今日异常数
* 待处理申诉数
* 当前在线教师数
* 当前在线学生数

示例：

| 指标    | 数值    |
| ----- | ----- |
| 今日任务  | 128   |
| 参与学生  | 8,642 |
| 平均完成率 | 92.6% |
| 异常记录  | 213   |
| 待审核申诉 | 46    |

---

### 2. 风险预警区

展示系统自动识别出的风险信息。

风险类型包括：

* 某学院异常率明显升高
* 某班级连续多日完成率低
* 某教师发布任务后长期未处理异常
* 某学生多次定位异常
* 实习打卡存在安全风险上报
* 某类任务异常集中

示例：

> 软件学院今日课堂签到异常率为 12.4%，高于近7日平均水平，请关注。

> 2024级软件技术1班近3日查寝完成率低于80%，建议提醒辅导员跟进。

---

### 3. 今日任务运行情况

以列表或图表展示：

* 任务名称
* 所属学院
* 发布教师
* 打卡类型
* 参与人数
* 完成率
* 异常数
* 当前状态

操作按钮：

* 查看详情
* 查看异常
* 联系负责人

---

### 4. 快捷入口

管理员高频操作入口：

* 新增教师
* 导入学生
* 创建班级
* 配置打卡类型
* 配置规则模板
* 查看异常申诉
* 导出数据报表

# 八、页面五：打卡类型管理

这是管理员端非常核心的页面。

## 页面名称

打卡类型管理

## 页面目标

统一管理平台支持的打卡类型，为教师端创建任务提供可选类型。系统不应把打卡模式固定死，而应支持管理员维护预设类型和自定义类型。

---

## 打卡类型列表

系统预设类型包括：

| 类型    | 适用场景          |
| ----- | ------------- |
| 查寝类   | 晚间归寝、安全确认     |
| 课堂类   | 上课签到、课堂考勤     |
| 实习类   | 校外实习、岗位实践     |
| 活动类   | 班会、讲座、志愿活动    |
| 会议类   | 学生干部会议、主题教育会议 |
| 早操类   | 晨练、晨读、早操      |
| 返校类   | 假期返校、离校返校确认   |
| 安全上报类 | 实习、返乡、外出安全确认  |
| 自定义类  | 特殊管理场景        |

---

## 打卡类型字段

每个打卡类型包含：

* 类型名称
* 类型编码
* 适用场景
* 默认图标
* 默认说明
* 是否启用
* 可使用角色
* 可使用范围
* 默认规则模板
* 是否允许教师自定义

---

## 打卡类型操作

支持：

* 新增类型
* 编辑类型
* 启用 / 停用
* 设置默认规则
* 设置可用教师角色
* 设置适用组织范围
* 查看使用记录
* 删除未使用类型

---

## 示例

### 类型名称

课堂签到

### 类型说明

适用于课程教学场景下的学生签到，可配置动态码、定位、人脸识别等校验方式。

### 默认规则

* 时间规则：课程开始前10分钟至课程开始后5分钟
* 地点规则：教学楼范围
* 校验方式：动态码 + 定位
* 审核规则：异常才审核

---

# 九、页面六：规则模板管理

## 页面名称

规则模板管理

## 页面目标

管理员统一维护常用打卡规则模板，教师端可以直接使用模板创建任务，降低教师配置成本，同时保证管理规范。

---

## 模板类型

### 系统模板

由管理员维护，教师可使用但不可随意修改原模板。

### 学院模板

由学院管理员维护，仅本学院教师可使用。

### 教师个人模板

教师端创建，管理员可查看但不一定能编辑。

---

## 模板列表字段

* 模板名称
* 适用打卡类型
* 适用组织
* 创建人
* 使用次数
* 最近使用时间
* 状态

---

## 模板配置内容

规则模板应包含以下模块：

### 1. 时间规则

* 单次打卡
* 周期打卡
* 多时间段打卡
* 上班 / 下班打卡
* 允许迟到
* 允许补卡
* 补卡截止时间

### 2. 地点规则

* 不限制地点
* 校园范围
* 教学楼范围
* 宿舍范围
* 实习单位范围
* 自定义地点
* 定位容差范围

### 3. 身份校验规则

* 无需校验
* 人脸识别
* 动态码
* 二维码
* 教师现场确认
* 多重校验

### 4. 提交内容规则

* 文字说明
* 图片上传
* 实习日志
* 安全状态
* 自定义表单
* 附件上传

### 5. 审核规则

* 自动通过
* 异常才审核
* 全部人工审核
* 补卡需审核
* 申诉需审核

### 6. 提醒规则

* 开始前提醒
* 截止前提醒
* 未完成提醒
* 异常提醒
* 审核结果通知

---

## 模板操作

* 新增模板
* 编辑模板
* 复制模板
* 启用 / 停用
* 设置默认模板
* 分配适用范围
* 查看使用情况

# 十二、页面九：数据分析

## 页面名称

数据分析

## 页面目标

---

## 统计维度

### 1. 总体数据

* 总任务数
* 总参与学生数
* 总完成率
* 总异常率
* 申诉通过率
* 教师任务使用率

### 2. 按组织统计

* 学院完成率排行
* 专业完成率排行
* 班级完成率排行
* 班级异常率排行

---

### 3. 按打卡类型统计

* 查寝类完成率
* 课堂类完成率
* 实习类完成率
* 活动类完成率
* 早操类完成率
* 自定义类型完成率

---

### 4. 按学生画像统计

不建议做过度标签化，但可以做管理辅助统计：

* 连续未打卡学生

* 长期高完成率学生
* 实习安全风险学生
* 课堂出勤偏低学生

---

### 5. 按教师使用情况统计

* 教师发布任务数
* 教师处理异常数量
* 教师平均处理时长
* 教师使用模板情况
* 教师任务完成率

---

* 折线图：趋势变化

* 饼图：打卡类型占比
* 热力图：异常高发时段

## 数据导出

# 十三、页面十：AI能力配置

## 页面名称

AI能力配置

## 页面目标

管理系统中的AI能力开关、识别规则、风险预警规则和智能分析能力。

---

## AI能力模块

### 1. 人脸识别配置

支持设置：

* 是否启用人脸识别
* 识别失败重试次数
* 识别通过阈值
* 是否允许人工复核
* 人脸数据授权说明

---

### 2. 定位异常识别

支持设置：

* 定位偏差范围
* 是否允许定位漂移容忍
* 是否启用多次定位校验
* 是否启用异常位置预警
* 是否允许学生提交定位说明

---

### 3. 疑似代签识别

可基于以下规则判断：

* 同一设备多人打卡
* 短时间内多人同地点异常打卡
* 动态码错误多次尝试
* 定位与任务地点明显不符
* 人脸识别失败后仍提交

---

### 4. 实习安全风险识别

可识别：

* 长期未提交实习日志
* 多次不在实习单位范围
* 学生主动上报风险
* 实习日志出现风险关键词
* 下班打卡长期缺失

---

### 5. 智能提醒配置

支持：

* 未打卡自动提醒
* 异常自动提醒教师
* 高风险自动提醒管理员
* 长期异常自动生成关注名单

---

## 注意事项

AI能力配置页面需要明确：

* AI只作为辅助判断
* 最终处理结果应由教师或管理员审核
* 涉及学生权益的判断应保留申诉通道
* 涉及人脸和定位数据应有隐私授权说明

# 十六、管理员端角色权限设计

管理员端需要支持多级权限，否则数据容易越权。

## 角色建议

| 角色      | 权限范围              |
| ------- | ----------------- |
| 系统超级管理员 | 管理全平台配置、用户、权限和数据  |
| 学校管理员   | 查看全校数据，管理学校级配置    |
| 学院管理员   | 管理本学院组织、教师、学生和数据  |
| 学工管理员   | 重点管理学生安全、查寝、异常和申诉 |
| 教务管理员   | 重点查看课堂考勤、课程班数据    |
| 实习管理员   | 重点查看实习打卡、日志、安全风险  |
| 数据审计员   | 只读查看数据和日志         |

---

## 权限控制维度

权限不只控制菜单，还要控制数据范围。

建议从以下维度设计：

1. 菜单权限
   能不能进入某个页面。

2. 操作权限
   能不能新增、编辑、删除、导出、审核。

3. 数据范围权限
   能看全校、本学院、本专业、本班级，还是只读部分数据。

4. 敏感信息权限
   能不能查看定位、人脸、联系方式、申诉详情。

5. 导出权限
   能不能导出学生数据和统计报表。

# 十九、管理员端设计重点

## 1. 管理员端要服务“平台治理”

学生端和教师端关注具体任务，管理员端关注整体规则、权限和数据。管理员端不应该只是教师端的放大版，而应该承担平台治理作用。

---

## 2. 打卡类型必须可配置

由于系统未来不局限于查寝、上课、实习三种模式，因此管理员端必须有“打卡类型管理”页面。管理员可以新增、停用、配置不同打卡类型。

---

## 3. 规则模板必须统一管理

教师端允许自定义，但完全放开会导致管理标准不统一。管理员端需要提供系统模板和学院模板，让教师在规范范围内灵活使用。

---

## 4. 权限管理必须细致

不同教师、学院、部门能看到的数据范围不同。特别是定位、人脸、申诉等敏感信息，需要严格控制查看和导出权限。

---

## 5. 数据分析要服务思政和管理

管理员端的数据分析不仅是“谁没打卡”，还要支撑：

* 安全教育
* 诚信教育
* 学风建设
* 实习过程监管
* 班级治理
* 学生发展支持

---

## 6. AI能力要可控

AI识别、风险预警、异常判断都应该可配置、可解释、可人工复核，不能让AI直接决定学生处理结果。

---

# 二十、最终建议

管理员端建议定位为：

> 面向学校和学院管理人员的 Web 管理后台，负责维护组织、用户、班级、分组、打卡类型、规则模板和权限体系，监管全校或学院范围内的打卡任务运行情况，集中查看异常与申诉，并通过数据分析和AI辅助预警，为学生安全管理、学风建设、诚信教育和实习过程管理提供决策支持。

管理员端第一阶段应优先建设七个核心模块：

1. 组织管理
2. 用户管理
3. 班级与分组管理
4. 打卡类型管理
5. 规则模板管理
6. 任务监管
7. 异常与申诉管理

第二阶段再增强：

1. 数据分析大屏
2. AI风险预警
3. 导出审批
4. 操作审计
5. 多学院数据对比
6. 智能成长画像
