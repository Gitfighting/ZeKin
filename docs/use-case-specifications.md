# ZeKin 72 小时 MVP 用例规约

## 1. 规约模板

每个用例以业务目标为中心描述，而不是以按钮或接口为中心描述。用例必须说明参与者、触发条件、前置条件、主成功场景、扩展场景、后置条件、业务规则和验收标准。

## UC-01 用户登录

| 项 | 内容 |
---|---|
| 用例名称 | 用户登录 |
| 主要参与者 | 学生、辅导员/教师、管理员 |
| 业务目标 | 用户以合法身份进入系统，并获得与角色匹配的功能入口 |
| 触发条件 | 用户打开系统并提交账号密码 |
| 前置条件 | 用户账号已存在且状态可用 |
| 后置条件 | 系统建立登录态，跳转到对应工作台 |
| 优先级 | Must |

主成功场景：

1. 用户打开登录页。
2. 用户输入账号和密码。
3. 系统校验账号是否存在。
4. 系统校验密码是否正确。
5. 系统识别用户角色。
6. 系统生成登录凭证。
7. 系统将学生引导至学生首页，将教师引导至审核工作台，将管理员引导至统计页。

扩展场景：

- 3a. 账号不存在：系统提示“账号或密码错误”。
- 4a. 密码错误：系统提示“账号或密码错误”。
- 5a. 账号被禁用：系统提示“账号不可用，请联系管理员”。

业务规则：

- BR-LOGIN-01：登录失败提示不得泄露账号是否真实存在。
- BR-LOGIN-02：登录成功后，用户只能访问角色授权范围内的页面。

验收标准：

- 给定有效学生账号，登录后进入学生首页。
- 给定有效教师账号，登录后进入教师审核页。
- 给定错误密码，系统拒绝登录并显示明确提示。

## UC-02 提交打卡

| 项 | 内容 |
---|---|
| 用例名称 | 提交打卡 |
| 主要参与者 | 学生 |
| 辅助参与者 | 定位服务 |
| 业务目标 | 学生在规定业务场景下提交可追溯的打卡记录 |
| 触发条件 | 学生点击“去打卡”或进入打卡页 |
| 前置条件 | 学生已登录；学生今日该类型尚未提交有效打卡 |
| 后置条件 | 系统生成一条待教师审核的打卡记录 |
| 优先级 | Must |

主成功场景：

1. 学生进入今日打卡页。
2. 系统展示查寝、上课、实习三类打卡入口。
3. 学生选择打卡类型。
4. 学生填写打卡内容。
5. 系统请求定位服务获取经纬度。
6. 学生可选择上传照片或保留照片占位。
7. 学生提交打卡。
8. 系统校验内容、类型和重复打卡规则。
9. 系统保存打卡记录，状态为“待审核”。
10. 系统刷新今日状态，并提示“提交成功，等待教师审核”。

扩展场景：

- 5a. 定位失败：系统允许继续提交，但将定位状态标记为异常。
- 8a. 内容为空：系统提示学生补充打卡内容。
- 8b. 同日同类型已提交：系统拒绝提交并提示“今日该类型已打卡”。
- 9a. 保存失败：系统提示稍后重试，不改变今日状态。

业务规则：

- BR-CHECKIN-01：同一学生、同一自然日、同一打卡类型只允许一条有效记录。
- BR-CHECKIN-02：内容长度为 10-300 字。
- BR-CHECKIN-03：定位异常不阻断提交，但必须展示给教师。

验收标准：

- 学生可以完成查寝打卡。
- 重复提交同类打卡时被系统拦截。
- 定位失败时仍可提交，并在教师端看到异常标识。

## UC-03 查看今日状态与历史

| 项 | 内容 |
---|---|
| 用例名称 | 查看今日状态与历史 |
| 主要参与者 | 学生 |
| 业务目标 | 学生确认今日任务完成情况，并追踪教师审核结果 |
| 触发条件 | 学生进入首页或历史页 |
| 前置条件 | 学生已登录 |
| 后置条件 | 学生理解自己的打卡状态 |
| 优先级 | Must |

主成功场景：

1. 学生进入首页。
2. 系统查询今日三类打卡状态。
3. 系统展示每类状态和下一步操作。
4. 学生查看最近打卡记录。
5. 系统展示每条记录的打卡类型、提交时间、审核状态和评语摘要。

扩展场景：

- 2a. 今日无打卡记录：系统显示“待打卡”。
- 5a. 记录为空：系统展示空状态和“立即打卡”入口。

验收标准：

- 首页能清晰看到三类状态。
- 被驳回记录能看到教师评语。

## UC-04 教师审核打卡

| 项 | 内容 |
---|---|
| 用例名称 | 教师审核打卡 |
| 主要参与者 | 辅导员/教师 |
| 业务目标 | 教师快速处理班级打卡记录，形成管理闭环 |
| 触发条件 | 教师进入审核工作台 |
| 前置条件 | 教师已登录；教师拥有班级权限 |
| 后置条件 | 打卡记录更新为通过或驳回 |
| 优先级 | Must |

主成功场景：

1. 教师进入审核工作台。
2. 系统默认展示今日、所管班级、待审核打卡。
3. 教师按班级、日期、类型或状态筛选。
4. 教师打开某条打卡详情。
5. 系统展示学生信息、内容、定位状态、照片占位、提交时间。
6. 教师选择通过或驳回。
7. 如驳回，教师填写原因。
8. 系统保存审核记录，并更新打卡审核状态。
9. 系统刷新列表和统计数字。

扩展场景：

- 3a. 无记录：系统展示空状态。
- 4a. 教师查看非授权班级：系统拒绝访问。
- 7a. 驳回原因为空：系统提示填写原因。

业务规则：

- BR-REVIEW-01：教师只能审核所管班级。
- BR-REVIEW-02：驳回必须填写原因。
- BR-REVIEW-03：审核结果必须同步给学生端。

验收标准：

- 教师能筛选并审核待处理记录。
- 驳回记录在学生端可见。
- 非授权教师不能查看其他班级记录。

## UC-05 查看统计概览

| 项 | 内容 |
---|---|
| 用例名称 | 查看统计概览 |
| 主要参与者 | 辅导员/教师、管理员 |
| 业务目标 | 管理人员用最短时间掌握今日打卡完成度和异常情况 |
| 触发条件 | 用户进入统计页 |
| 前置条件 | 用户已登录且具备统计权限 |
| 后置条件 | 系统展示当前范围内的关键管理指标 |
| 优先级 | Must |

主成功场景：

1. 用户进入统计概览页。
2. 系统识别用户角色和数据范围。
3. 教师默认查看所管班级，管理员默认查看全校。
4. 系统聚合今日应打卡人数、已打卡人数、打卡率、待审核数、异常数。
5. 页面展示指标卡片和趋势占位。
6. 用户根据统计结果进入审核列表处理异常。

扩展场景：

- 2a. 用户无统计权限：系统拒绝访问。
- 4a. 暂无数据：系统展示空状态和演示数据导入提示。

验收标准：

- 教师只能看到所管班级统计。
- 管理员可以看到全局统计。
- 统计指标能引导教师进入待审核列表。
