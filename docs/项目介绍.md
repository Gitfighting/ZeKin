# 知勤 · 智慧考勤管理平台

> 项目介绍（提炼自 [打卡任务系统设计方案](./checkin-task-template-design.md)）

---

## 1. 项目概述

**知勤** 是一套面向高校与实训场景的智慧考勤管理系统，采用「**班级群组 + 任务模板 + 可组合签到**」的产品模型，类似钉钉考勤的核心思路：教师向班级发布打卡任务，学生在规则窗口内完成签到，管理员在 Web 端进行组织、人员与数据监管。

系统覆盖 **查寝、课堂、实习** 等典型场景，但不在代码层按场景分叉——差异全部收敛到 **模板配置**（时间窗口、签到方式组合及参数），由统一的校验流水线动态执行。

### 技术架构

| 模块 | 说明 |
|------|------|
| `backend/` | FastAPI 后端，打卡校验流水线、任务发布、消息通知等 |
| `admin-web/` | Vue 3 + Element Plus 管理端（组织、人员、任务监管、数据大屏） |
| `mini-app/` | uni-app 微信小程序（教师发布任务 / 学生动态打卡向导） |
| `docs/` | 设计文档与 API 契约 |

---

## 2. 要解决的核心问题

旧方案将「查寝 / 课程 / 实习」作为顶层业务分类，导致 seed、前端与后端逻辑各自分叉。**实际上三类场景形态一致**：

> 教师向班级群组发布打卡任务 → 群内学生收到通知 → 按规则完成打卡

「查寝 / 课程 / 实习」应仅是 **班级名称或模板的业务标签**，用于展示与筛选，**不应驱动不同代码分支**。

---

## 3. 核心概念

| 概念 | 说明 |
|------|------|
| **班级群组（Group）** | 任务发布与接收的基本单位，如「3 号寝室班」「软工导论 A202 班」「XX 公司实习班」 |
| **任务模板（TaskTemplate）** | 教师预设的可复用规则组合（时间、签到方式、提醒等） |
| **打卡任务（CheckinTask）** | 基于模板创建并发布到群组的具体实例，发布时冻结 `rules_snapshot` |
| **任务周期** | **一次任务**（临时签到）或 **定时任务**（每日/每周循环，教师只配置一次） |
| **签到方式** | 教师自由勾选、自由组合的验证手段，由插件化 Verifier 执行 |

---

## 4. 设计原则

1. **群组优先** — 一切任务以 `Group` 为发布单位，学生通过群组成员关系接收任务。
2. **模板驱动差异** — 场景差异收敛到 `TaskTemplate.rules`，代码零分叉。
3. **签到方式可组合** — 至少启用一种，全部通过才算成功；顺序由 `order` 配置。
4. **一次 vs 循环** — 任务周期是任务属性，不是场景属性。
5. **消息触达** — 任务发布后对群内学生推送通知。
6. **插件化扩展** — 每种签到方式实现为独立 `CheckinVerifier`，新增方式无需改动核心流程。

---

## 5. 支持的签到方式

教师创建任务时通过多选框勾选，可单选也可任意组合：

| 方式 | 标识 | 典型场景 |
|------|------|----------|
| 人脸识别 | `face` | 查寝、课程、实习身份核验 |
| 地理位置 | `location` | GPS + 地理围栏（Haversine） |
| 二维码签到 | `qr_code` | 课堂到场、防代签（动态刷新二维码） |
| 附件上传 | `attachment` | 实习日报、情况说明、现场照片 |
| 手势签到 | `gesture` | 快速点名、轻量趣味签到 |

**组合示例：**

| 场景 | 启用方式 | 说明 |
|------|----------|------|
| 晚间查寝 | `face` + `location` | 先人脸，再校验是否在宿舍围栏内 |
| 课堂签到 | `qr_code` + `face` | 扫教师投屏二维码，再拍照人脸 |
| 实习日报 | `face` + `location` + `attachment` | 人脸 + 定位 + 上传工作日志 |

时间窗口校验 **始终执行**，不属于可选签到方式。

## 6. 校验流水线（责任链 + 插件化）

学生提交打卡后，后端 `CheckinPipeline` 根据任务 `rules_snapshot.verificationRule` **动态组装**校验步骤：

```
1. TimeWindowVerifier（时间窗口，固定）
2. 按 order 依次执行启用的 Verifier：
   Face → Location → QrCode → Attachment → Gesture
3. 汇总 verification_results，写入 CheckinRecord
```

前端小程序按同一 `methods` + `order` **动态渲染步骤向导**（如 Step 1/3 人脸 → Step 2/3 定位 → Step 3/3 附件），分步收集后 **一次 POST 提交**，后端全量校验防绕过。

## 7. 角色与端能力

### 管理员（admin-web）

- 组织管理、人员管理（学生 / 教师）
- 班级与分组、任务监管、考勤异常与申诉
- 数据工作台与数据大屏

### 教师（mini-app）

- 管理班级群组，选用模板创建任务
- 勾选签到方式并配置参数（围栏半径、二维码刷新间隔等）
- 发布任务并通知群成员；二维码投屏签到
- 查看打卡统计，处理异常

### 学生（mini-app）

- 接收群组内已发布任务与消息
- 按任务配置完成动态打卡向导
- 查看打卡记录，提交异常申诉与日报（attachment 方式）

## 8. 三种场景配置对照

差异 **仅在 JSON 配置**，代码路径完全相同：

| 配置项 | 3 号寝室班 | 软工导论 A202 班 | XX 实习班 |
|--------|------------|------------------|-----------|
| 任务周期 | 每日循环 | 一次任务 | 每日循环 |
| 签到方式 | 人脸 + 定位 | 二维码 + 人脸 | 人脸 + 定位 + 附件 |
| 时间窗口 | 21:30–23:00 | 08:00–08:15 | 08:30–10:00 |
| 围栏半径 | 300 m | 100 m | 500 m |

## 10. 设计模式总结

| 维度 | 旧做法 | 本方案 |
|------|--------|--------|
| 顶层分类 | 查寝 / 课程 / 实习 三套模块 | **班级群组 + 任务模板** |
| 签到方式 | 按场景硬编码 | **自由勾选、自由组合** |
| 差异表达 | type_id 分支代码 | 同一 Pipeline + 不同 methods 配置 |
| 实习日报 | 独立 daily_report 模块 | 归入 **attachment** 签到方式 |
| 扩展性 | 加场景需改代码 | 新增 Verifier 插件即可 |

**设计模式：** 模板方法 + 责任链 + 策略（Verifier）

## 12. 本地快速启动

```powershell
# 基础设施
docker compose up -d

# 后端
just backend-dev

# 管理端
just admin-dev

# 小程序
just mini-dev
```

管理端默认访问开发地址（如 `http://localhost:5173`），API 默认 `http://localhost:8000/api`。
