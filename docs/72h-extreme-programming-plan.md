# ZeKin 72 小时极限开发计划

## 1. 原则

执行原则：

- 只做 Must，Should 只在核心闭环稳定后进入。
- 数据库、接口、页面先跑通，再优化细节。
- 所有成员每天至少两次集成，避免最后 6 小时才联调。
- 任何超过 2 小时无进展的问题必须降级、绕过或换人。
- AI、人脸、复杂导出不阻塞 MVP。

## 2. 时间盒

| 时间 | 目标 | 交付物 |
---|---|---|
| H0-H6 | 需求冻结 | 用例、图、原型、验收脚本 |
| H6-H12 | 项目骨架 | 后端、前端、数据库最小启动 |
| H12-H24 | 认证与基础数据 | 登录、角色、演示数据 |
| H24-H36 | 学生打卡闭环 | 今日状态、提交打卡、历史 |
| H36-H48 | 教师审核闭环 | 审核列表、详情、通过/驳回 |
| H48-H60 | 统计与联调 | 指标概览、权限、异常处理 |
| H60-H66 | 四环境准备 | 本地、服务器、UAT、生产测试配置 |
| H66-H72 | 验收与修复 | 演示脚本、缺陷修复、交付说明 |

## 4. Git 规则

- `main`：仅放可验收版本。
- `develop`：72 小时集成分支。
- `feat/backend-minimal`：后端。
- `feat/admin-review`：教师/管理端。
- `feat/student-checkin`：学生端。
- `docs/uml-requirements`：需求建模。

合并规则：

- 每 12 小时至少合并一次到 `develop`。
- 接口字段变化必须先通知前端成员。
- 任何破坏启动的提交必须 30 分钟内修复或回滚。

## 5. 验收脚本

1. 管理员登录，查看统计页。
2. 学生 A 登录，查看今日三类状态。
3. 学生 A 提交查寝打卡。
4. 学生 A 重复提交查寝，被系统拒绝。
5. 教师登录，筛选今日查寝待审核。
6. 教师查看学生 A 详情并通过。
7. 学生 A 刷新历史，看到审核通过。
8. 学生 B 提交定位异常记录。
9. 教师驳回学生 B 并填写原因。
10. 统计页待审核数、异常数发生变化。
