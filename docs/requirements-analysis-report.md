# ZeKin AI 思政打卡系统 72 小时 MVP 需求分析报告

版本：V2.0  
日期：2026-06-22  
项目周期：3 天 × 24 小时 = 72 小时极限编程  
建模方法：业务叙事 + UML 需求分析 + PlantUML 图形化表达  
技术约束：Python + FastAPI + Vue + MySQL，前后端分离  
团队成员：王韩韵、赵耀、胡钊炫、华心仪

## 1. 建模依据与方法

UML 是面向软件与系统制品的可视化、规格化、构造和文档化建模语言；本报告采用 UML 作为需求分析语言，用图形描述业务边界、角色目标、流程协作和核心领域对象。用例规约用于描述“用户想达成的业务目标”，活动图用于描述业务流程，顺序图用于描述系统对象协作，时序图用于描述时间窗口和状态变化，分析类图用于沉淀领域概念。

本报告采用以下建模链路：

1. 业务痛点：识别学校、教师、学生在查寝、上课、实习管理中的真实问题。
2. 业务目标：定义 72 小时内必须闭环的最小业务价值。
3. 用例建模：从角色视角定义系统对外提供的能力。
4. 用例规约：为每个核心用例定义前置条件、主成功场景、扩展场景、业务规则和验收标准。
5. 行为建模：用活动图、顺序图、时序图解释关键流程。
6. 领域建模：用分析类图明确核心实体、边界类、控制类及关系。
7. 原型建模：用低保真样图表达信息架构和关键页面布局。

参考资料包括 OMG UML 2.5.1、PlantUML 官方语法、Visual Paradigm 用例规约与 UML 图实践。链接见本文末尾“参考资料”。

## 2. 项目背景

高校学生日常管理存在三个高频而低效的业务场景：

- 查寝：辅导员夜间人工点名，耗时高，记录分散，异常追踪慢。
- 上课：纸质签到或口头点名占用教学时间，代签风险高。
- 实习：学生分散在不同城市和企业，学校难以及时掌握学生安全状态和实习动态。

ZeKin 的业务定位不是“签到小工具”，而是“学生日常到岗与思想动态管理系统”。它要解决的是：

- 学生是否在规定时间完成打卡。
- 打卡内容是否具备基本合规性和可审核性。
- 教师是否能快速发现未打卡、异常打卡和待审核记录。
- 学校是否能通过统计数据掌握班级、课程、实习管理情况。

## 3. 72 小时 MVP 策略

原需求包含 AI 审核、人脸识别、补签、提醒、积分、导出、复杂统计等能力。按照 72 小时极限编程约束，MVP 必须收敛到“核心闭环”，否则团队会被集成风险拖垮。

### 3.1 MVP 成功定义

72 小时后，系统必须能演示以下完整业务闭环：

1. 学生登录后看到今日三类打卡状态。
2. 学生选择查寝、上课或实习，提交文字、定位和可选照片信息。
3. 系统阻止同一学生同日同类型重复打卡。
4. 教师按班级、日期、类型查看学生打卡列表。
5. 教师对打卡记录进行通过或驳回，并填写评语。
6. 学生可看到审核状态。
7. 教师或管理员看到班级打卡率、待审核数、异常数等统计概览。

### 3.2 进入 MVP 的功能

| 优先级 | 功能 | 业务理由 |
|---|---|---|
| Must | 登录与角色识别 | 没有身份就无法区分学生、教师、管理员 |
| Must | 三类打卡提交 | 覆盖查寝、上课、实习三大核心场景 |
| Must | 今日状态与历史记录 | 学生需要确认自己是否完成任务 |
| Must | 教师审核列表 | 教师需要处理班级考勤与异常 |
| Must | 审核通过/驳回 | 形成管理闭环 |
| Must | 基础统计 | 支撑验收演示和管理价值表达 |
| Should | 定位记录 | 先记录位置，为后续范围校验预留 |
| Should | 照片上传占位 | 先保存 URL 或占位字段，不做复杂文件服务 |

### 3.3 不进入 72 小时核心实现

| 功能 | 处理方式 | 原因 |
|---|---|---|
| AI 内容审核 | 作为扩展点，先用 `ai_status=pending` | 外部接口不确定，72 小时内不做阻塞项 |
| 人脸识别 | 仅预留字段 | SDK、隐私、准确率都需要额外验证 |
| 补签审批 | 下一迭代 | 增加流程复杂度 |
| 推送通知 | 下一迭代 | 依赖移动端通道或浏览器通知策略 |
| 积分等级 | 下一迭代 | 非核心管理闭环 |
| Excel 导出 | 可选增强 | 若时间不足，用统计页面替代 |

## 4. 业务角色

| 角色 | 业务目标 | 权限范围 |
|---|---|---|
| 学生 | 在规定场景和时间内完成打卡，查看自己的审核结果 | 只能创建和查看本人打卡记录 |
| 辅导员/教师 | 查看所管班级打卡情况，审核学生记录，发现异常 | 查看所管班级学生数据，审核打卡 |
| 管理员 | 查看全局统计，配置基础数据，支撑验收演示 | 查看全系统数据，管理用户和基础配置 |
| 定位服务 | 为打卡提供经纬度 | 外部服务，不保存业务数据 |

## 6. 业务规则

| 编号 | 规则 |
---|---|
| BR-01 | 一个学生同一自然日、同一打卡类型只能提交一次有效打卡 |
| BR-02 | 打卡类型仅允许 `查寝`、`上课`、`实习` 三类 |
| BR-03 | 打卡内容必须填写，72 小时 MVP 中建议限制为 10-300 字 |
| BR-04 | 定位失败不阻断提交，但记录为“定位异常”，供教师审核参考 |
| BR-05 | 教师只能审核自己所管班级学生的打卡 |
| BR-06 | 已审核记录再次审核时必须覆盖保存最新审核人、评语和时间 |
| BR-07 | 管理员可查看全局统计，教师只能查看所管班级统计 |
| BR-08 | 所有写操作必须记录创建时间、更新时间和操作人 |

## 7. 功能需求

### 7.1 学生端

学生打开系统后的第一任务不是浏览信息，而是确认“今天还有哪些事没完成”。因此学生端首页必须将“今日状态”放在首屏，打卡入口必须明显，历史记录只作为确认与追溯。

| 需求 ID | 功能 | 验收标准 |
---|---|---|
| FR-S-01 | 登录 | 输入账号密码后进入学生首页 |
| FR-S-02 | 今日状态 | 展示查寝、上课、实习三个状态：未开始、待打卡、已提交、已通过、被驳回 |
| FR-S-03 | 提交打卡 | 支持选择类型、填写内容、采集定位、提交 |
| FR-S-04 | 防重复 | 同日同类型重复提交时给出明确提示 |
| FR-S-05 | 历史记录 | 展示本人打卡时间、类型、教师审核结果 |

### 7.2 教师端

教师端的业务价值在于“快速处理异常”。页面不应让教师逐条翻找，而应优先展示待审核、未打卡、被驳回等状态。

| 需求 ID | 功能 | 验收标准 |
---|---|---|
| FR-T-01 | 教师登录 | 登录后进入审核工作台 |
| FR-T-02 | 筛选班级打卡 | 按班级、日期、类型、状态筛选 |
| FR-T-03 | 查看详情 | 查看学生姓名、班级、内容、定位、照片占位、提交时间 |
| FR-T-04 | 审核通过 | 记录审核结果为通过 |
| FR-T-05 | 审核驳回 | 记录驳回原因，学生端可见 |

### 7.3 管理与统计

72 小时 MVP 的统计不是 BI 系统，而是验收演示和教师管理所需的最小指标。

| 需求 ID | 功能 | 验收标准 |
---|---|---|
| FR-A-01 | 统计概览 | 展示今日应打卡人数、已打卡人数、打卡率、待审核数、异常数 |
| FR-A-02 | 角色范围 | 教师看班级，管理员看全局 |
| FR-A-03 | 演示数据 | 提供可用于验收演示的数据样本 |

## 8. 非功能需求

| 分类 | 72 小时 MVP 标准 |
---|---|
| 性能 | 常规列表接口在演示数据量下 500ms 内返回 |
| 安全 | 密码不明文存储；所有业务接口必须登录访问 |
| 可用性 | 本地、服务器、UAT、生产测试四环境均可启动或可演示 |
| 可维护性 | 前后端分离；接口路径、字段、状态枚举统一 |
| 可追溯 | 核心操作保留创建时间、审核时间、审核人 |
| 可演示 | 系统内置一组学生、教师、管理员和打卡记录样例 |

## 10. 分析类模型说明

分析类采用 BCE 方法：

- Boundary：直接面向用户或外部系统的边界对象，例如学生打卡页面、教师审核页面、定位服务适配器。
- Control：承载用例流程控制，例如打卡控制器、审核控制器、统计控制器。
- Entity：长期保存的业务对象，例如用户、打卡记录、审核记录、班级、课程、地点规则。

这种拆分可以让 72 小时开发不混乱：前端同学围绕 Boundary 工作，后端同学围绕 Control 与 Entity 工作，产品同学围绕用例规约和验收标准工作。

## 11. 原型设计

原型样图：

- 学生打卡页：`prototypes/student-checkin.svg`
- 教师审核页：`prototypes/teacher-review.svg`
- 管理统计页：`prototypes/admin-dashboard.svg`

原型设计原则：

- 学生端优先减少操作步数，首屏展示今日状态和一键打卡。
- 教师端优先处理待审核队列，突出筛选、状态和审核动作。
- 管理端优先展示关键指标，避免 72 小时内做复杂报表。

| 环境 | 72 小时验收方式 | 通过标准 |

| 本地开发环境 | 每位成员本机启动前后端与数据库 | 可完成登录、打卡、审核、统计 |
