# 知勤打卡系统 72 小时 MVP 需求精简与里程碑设计

版本：V1.0  
日期：2026-06-23  
表达风格：高中生能懂  
技术栈：Python + FastAPI + MySQL + Redis + 微信小程序 + Vue 管理端  
日志参考：`23-logging.md`、`23-logging-format.md`

## 一、项目一句话说明

知勤打卡系统可以理解成“学校版打卡小程序”。

学生每天用微信小程序完成查寝、上课、实习打卡；教师用微信小程序查看班级打卡并审核；管理员用 Web 管理端查看统计数据和系统情况。

72 小时 MVP 的目标不是做一个完美产品，而是先验证最核心流程是否走得通：

```text
学生打卡 -> 教师审核 -> 学生查看结果 -> 管理端查看统计
```

## 二、需求精简

### 2.1 功能剪减思路：ICE 优先级法

我们用 ICE 三个维度决定每个功能的优先级：

- Impact（影响力）：这个功能对用户价值有多大？
- Confidence（可行性）：我们 72 小时内有多大把握做出来？
- Ease（容易度）：实现起来有多难？

三个分数相加，分数越高，越应该进入 MVP。

| 功能模块 | I 影响力 | C 可行性 | E 容易度 | 总分 | MVP 决策 |
|---|---:|---:|---:|---:|---|
| 用户登录/注册 | 9 | 9 | 9 | 27 | 必做（P0） |
| 学生提交打卡 | 10 | 8 | 7 | 25 | 必做（P0） |
| 打卡历史查看 | 7 | 9 | 8 | 24 | 必做（P0） |
| 教师审核面板 | 9 | 7 | 6 | 22 | 简化做（P1） |
| 统计看板 | 6 | 7 | 5 | 18 | 简化做（P1） |
| AI 内容审核 | 5 | 4 | 3 | 12 | 不做（P2） |
| 人脸识别 | 3 | 3 | 2 | 8 | 不做（P2） |
| 语音打卡 | 2 | 2 | 2 | 6 | 不做（P2） |
| 积分/排行榜 | 4 | 3 | 3 | 10 | 不做（P2） |

### 2.2 MVP 剪减后的最终功能清单

经过 ICE 评分，72 小时 MVP 只保留 5 个功能。

| 序号 | 功能模块 | 负责人 | 功能说明 |
|---:|---|---|---|
| 1 | 用户登录/注册 | 赵耀 | 手机号注册 + JWT 认证，支持学生/教师/管理员三种角色 |
| 2 | 学生提交打卡 | 王韩韵 | 微信小程序提交文字心得 + 照片上传 + GPS 定位，支持查寝/上课/实习三种类型 |
| 3 | 打卡历史查看 | 胡钊炫 | 学生在微信小程序查看自己的打卡记录、审核状态和筛选 |
| 4 | 教师审核面板 | 华心仪 | 教师在微信小程序查看班级学生打卡列表，通过/拒绝并留评语（简化版） |
| 5 | 简易统计看板 | 华心仪 | 管理端 Web 展示打卡率、未打卡人数、连续打卡天数 |

### 2.3 为什么剪掉 AI 审核

这是 MVP 中最重要的产品决策。

- 对外部 API 依赖太重：72 小时内如果 AI API 出问题，整个打卡流程可能卡死。
- 审核结果不确定：AI 可能误判，学生和老师都需要额外解释，我们没时间调优。
- 教师审核已足够：MVP 阶段只要教师手动审核能走通，就能证明核心流程成立。

设计原则：

> MVP 的目标是验证核心流程可行，而不是一次性做一个完美产品。AI 审核放到 V1.1。

## 三、技术架构设计

### 3.1 最简分层架构

整个系统分为 6 层，每层只做一件事。

| 层级 | 负责 | 技术选型 |
|---|---|---|
| 学生端 | 学生提交打卡、查看历史和审核结果 | 微信小程序 |
| 教师端 | 教师查看班级打卡、通过/拒绝审核 | 微信小程序 |
| 管理端 | 管理员查看统计、系统状态和基础配置 | Vue 3 + Element Plus + Vite |
| 后端 API 层 | 接收请求，处理登录、打卡、审核、统计 | Python 3.11 + FastAPI |
| 数据存储层 | 长期保存用户、打卡记录、审核记录 | MySQL 8.0 |
| 缓存层 | 保存验证码、限流、短期统计缓存、重复提交锁 | Redis |
| 部署层 | 把项目托管到服务器，让别人能访问 | Docker Compose |

高中生可以这样理解：

- 微信小程序和 Web 是“门面”。
- FastAPI 是“服务台”。
- MySQL 是“长期档案柜”。
- Redis 是“临时便签本”。
- Docker Compose 是“一键启动按钮”。

### 3.2 数据库设计：仅 3 张 MySQL 表

原版需求有更多表，但 MVP 只保留 3 张核心表。

| 表名 | 存什么 | 关键字段 |
|---|---|---|
| `users` | 用户信息 | `id`, `username`, `password_hash`, `real_name`, `role(student/teacher/admin)`, `class_name`, `phone` |
| `checkins` | 打卡记录 | `id`, `user_id`, `type(dorm/class/internship)`, `content`, `photo_url`, `lat`, `lng`, `status(pending/approved/rejected)`, `created_at` |
| `reviews` | 审核记录 | `id`, `checkin_id`, `reviewer_id`, `action(approved/rejected)`, `comment`, `created_at` |

剪掉的表：

- `courses`：课程表。
- `allowed_locations`：定位白名单。
- `makeup_requests`：补签申请。

剪掉原因：

这些表有价值，但不影响 72 小时 MVP 的主流程验证。等 V1.1 再加更稳。

### 3.3 Redis 使用范围

Redis 不替代 MySQL。MySQL 负责长期保存，Redis 负责临时高速存取。

| 用途 | Key 示例 | 过期时间 | 说明 |
|---|---|---:|---|
| 验证码 | `sms:code:<phone>` | 5 分钟 | 注册/登录验证码，MVP 可先 mock |
| 接口限流 | `rate:<user_id>:<api>` | 60 秒 | 防止短时间重复提交 |
| 统计缓存 | `stats:overview:<date>:<class>` | 1-5 分钟 | 减少频繁聚合查询 |
| 重复打卡锁 | `lock:checkin:<user_id>:<type>:<date>` | 10 秒 | 防止重复点击导致并发提交 |
| Token 黑名单 | `auth:blacklist:<token_id>` | Token 剩余时间 | 用户退出或封禁后使 Token 失效 |

如果 72 小时内 Redis 联调风险变高，可以先保留封装和配置，让核心流程继续走 MySQL，不能让 Redis 阻塞主流程。

## 四、日志规范

日志规范参考用户提供的：

- `23-logging.md`
- `23-logging-format.md`

参考文件是 Rust/Axum/tracing 写法，本项目使用 Python + FastAPI，因此落地方式改成：

- Python 标准库 `logging`
- `TimedRotatingFileHandler` 按天切分日志
- FastAPI middleware 记录请求日志
- `request_id` 串联一次完整请求

### 4.1 标准日志格式

```text
[时间戳] [Emoji级别] [级别] [模块名] [文件名:行号] 日志内容
```

示例：

```text
2026-06-23 09:30:12.188 🟢 INFO  app.middleware.request_logger (request_logger.py:42) POST /api/checkins 200 37ms
2026-06-23 09:30:12.221 🔴 ERROR app.services.checkin_service (checkin_service.py:88) 创建打卡失败 user_id=12 error=DuplicateCheckin
```

### 4.2 日志级别

| 级别 | Emoji | 用途 |
|---|---|---|
| ERROR | 🔴 | 功能失败，例如数据库连接失败、写入失败 |
| WARN | 🟠 | 有风险但系统还能继续运行，例如 Redis 不可用降级 |
| INFO | 🟢 | 关键业务事件，例如服务启动、用户登录、打卡提交、审核完成 |
| DEBUG | 🟡 | 开发调试信息，例如请求参数摘要、SQL 耗时 |
| TRACE | ⚫ | 极细调用链，MVP 不建议默认开启 |

### 4.3 请求日志必须记录的字段

| 字段 | 示例 | 说明 |
|---|---|---|
| `request_id` | `req_abc123` | 每个请求唯一 ID，方便追踪 |
| `method` | `POST` | HTTP 方法 |
| `path` | `/api/checkins` | 请求路径 |
| `query` | `page=1` | 查询参数，敏感字段脱敏 |
| `status` | `200` | 响应状态码 |
| `latency_ms` | `37` | 请求耗时 |
| `user_id` | `12` | 已登录用户 ID，没有则为空 |
| `client` | `student-miniapp` | `student-miniapp` / `teacher-miniapp` / `admin-web` |

### 4.4 日志禁止记录

以下内容绝对不能写入日志：

- 明文密码。
- JWT Token 原文。
- 手机验证码。
- 身份证号等敏感个人信息。
- 图片二进制内容。

## 八、协作规范

- 每天 09:00 和 21:00 各开一次 15 分钟进度对齐会。
- 代码提交前必须自测，后端接口优先用 Postman 测试。
- 每个功能开一个 `feature/xxx` 分支，完成后合并到 `dev` 分支。
- 关键接口变更必须在群里通知所有人。
- 遇到阻塞超过 1 小时就沟通，不要一个人硬扛。
- 日志中不能出现密码、Token、验证码等敏感信息。

## 九、风险评估与应对策略

| 风险事项 | 可能影响 | 应对措施 |
|---|---|---|
| 前后端接口数据格式不一致 | 联调失败，页面无法显示数据 | 提前定义 API 响应规范，用 Postman 先测试后端 |
| 代码冲突 | 合并代码时出现冲突，修复浪费时间 | 每人负责不同模块，尽量避免修改同一个文件 |
| 环境不一致问题 | 本地能跑，服务器跑不起来 | 使用 Docker Compose 统一环境 |
| Redis 不可用 | 验证码、限流或缓存受影响 | 核心流程降级走 MySQL，不让 Redis 阻塞打卡 |
| 小程序真机调试问题 | 学生端/教师端演示受影响 | 准备微信开发者工具演示和测试二维码方案 |
| 进度延迟 | 某个功能花的时间超预期 | 准备降级方案：统计看板可以用简单列表代替图表 |
| 数据库性能 | 查询慢影响用户体验 | 为常用查询字段建索引，分页查询不过度加载 |

## 十、MVP 非功能需求

| 类别 | MVP 要求 | 具体指标 |
|---|---|---|
| 性能 | API 响应速度 | API 响应 < 1s |
| 安全 | 密码存储 + 身份认证 | 密码 bcrypt 哈希 + JWT Token，禁用明文 |
| 缓存 | Redis 可用但可降级 | Redis 异常不阻塞主流程 |
| 兼容性 | 小程序 + 主流浏览器 | 微信开发者工具、微信客户端、Chrome / Edge / Firefox 最新版 |
| 可用性 | 基本可用 | Docker 自动重启，MVP 不要求 99.9% 在线率 |
| 可观测性 | 结构化日志 | 请求日志包含 `request_id`, `method`, `path`, `status`, `latency_ms`, `user_id`, `client` |
