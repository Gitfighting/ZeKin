# 知勤打卡系统 UI 页面设计规范

版本：V1.0  
日期：2026-06-23  
适用范围：72 小时 MVP 的管理端 Web、学生端微信小程序、教师端微信小程序  
设计依据：`octo-code-DESIGN.md`、`micropost-DESIGN.md`  
技术约束：Python + FastAPI + MySQL + Redis + Vue 管理端 + 微信小程序

## 1. 设计目标

本 UI 设计服务于 72 小时极限 MVP，核心目标是让用户在最短路径内完成主流程：

```text
学生登录 -> 学生提交打卡 -> 教师审核 -> 学生查看结果 -> 管理员查看统计
```

MVP 阶段不追求复杂动效和完整后台能力，优先保证页面清晰、操作路径短、状态反馈明确、团队可分模块开发。

## 2. 设计系统选择

### 2.1 管理端 Web：采用 Octo Code 风格

管理端用户主要是管理员，使用场景偏桌面、偏数据查看、偏系统管理，因此采用 `octo-code-DESIGN.md` 的暗色、信息密集、工程化风格。

适配理由：

- 管理端需要同时查看统计、列表、状态、日志，信息密度要高。
- 深色界面适合长时间查看后台数据。
- Octo Code 的表格、侧边栏、标签、状态色更适合管理后台。

### 2.2 学生端和教师端小程序：采用 MicroPost 风格

学生和教师主要在手机上使用，操作高频、时间短、页面应轻量，因此采用 `micropost-DESIGN.md` 的浅色、卡片流、圆角按钮、移动端友好风格。

适配理由：

- 学生打卡像发一条短内容，天然适合卡片流。
- 教师审核像处理一组待办，适合紧凑列表和状态标签。
- MicroPost 的触控目标、卡片、状态 Chip 更适合微信小程序。

## 5. 公共组件设计

### 5.1 状态标签

| 状态 | 文案 | 颜色 | 使用位置 |
|---|---|---|---|
| 待审核 | 待审核 | 警告色 | 学生历史、教师审核列表、管理端表格 |
| 已通过 | 已通过 | 成功色 | 历史和审核结果 |
| 已拒绝 | 已拒绝 | 错误色 | 历史和审核结果 |
| 未打卡 | 未打卡 | 次文本色或错误色 | 数据看板、班级概览 |

小程序使用圆角胶囊 Chip；管理端使用小号 Label，圆角不超过 6px。

### 5.2 空状态

空状态文案必须短，不解释系统功能。

| 场景 | 文案 | 操作 |
|---|---|---|
| 无历史 | 暂无打卡记录 | 去打卡 |
| 无待审核 | 暂无待审核记录 | 刷新 |
| 无筛选结果 | 没有符合条件的记录 | 清空筛选 |
| 统计无数据 | 今日暂无数据 | 刷新 |

### 5.3 加载和错误

- 小程序列表加载：骨架屏 2-3 行卡片。
- 小程序提交中：按钮进入 loading，禁止重复提交。
- 管理端表格加载：表格骨架或 loading 遮罩。
- 网络错误：显示“网络异常，请稍后重试”，保留重试按钮。
- 权限错误：显示“当前账号无权访问该页面”，返回首页。

## 6. 学生端微信小程序页面设计

### 6.1 登录/注册页

目标：让学生快速进入系统。

布局：

```text
顶部：知勤打卡
副标题：今日打卡，从这里开始

输入区：
- 手机号
- 密码或验证码

按钮：
- 登录
- 注册账号

底部：
- 当前身份：学生
```

设计要点：

- 背景使用 `#F9FAFB`。
- 表单放在白色卡片中，圆角 12px。
- 主按钮使用蓝色胶囊按钮，高度不低于 40px。
- 登录失败时只提示“手机号或密码错误”，不要暴露账号是否存在。

### 6.2 今日打卡页

目标：学生打开小程序后立刻知道今天是否已打卡。

布局：

```text
顶部导航：知勤打卡

今日状态卡：
- 今日日期
- 状态：未打卡 / 待审核 / 已通过 / 已拒绝
- 主按钮：立即打卡 / 查看详情

打卡类型快捷入口：
- 查寝
- 上课
- 实习

最近记录：
- 最近 3 条打卡
```

交互：

- 点击“立即打卡”进入提交打卡页。
- 点击最近记录进入详情页。
- 如果今日已提交，主按钮改成“查看详情”，避免重复提交。

### 6.3 提交打卡页

目标：让学生在 1 分钟内完成打卡。

布局：

```text
顶部：提交打卡

类型选择：
[查寝] [上课] [实习]

内容输入：
请输入今日情况，最多 280 字
字数统计：0/280

照片上传：
[上传照片]

定位：
当前位置
重新获取

底部固定按钮：
提交打卡
```

设计要点：

- 类型选择使用 Filter Chip，选中态为蓝底白字。
- 内容输入最多 280 字，超过后禁止继续输入。
- 图片预览使用 16px 圆角。
- 底部提交按钮固定在安全区上方。

异常状态：

- 未选择类型：提示“请选择打卡类型”。
- 内容为空：提示“请填写打卡内容”。
- 定位失败：显示“定位失败，可重新获取”，不直接阻断页面。
- 提交失败：保留已输入内容，允许重新提交。

### 6.4 打卡历史页

目标：学生能快速看到自己的记录和审核结果。

布局：

```text
顶部：打卡历史

筛选 Chip：
[全部] [待审核] [已通过] [已拒绝]

记录卡片列表：
- 类型
- 状态
- 提交时间
- 内容摘要，最多两行
- 教师评语摘要
```

设计要点：

- 使用 MicroPost 卡片流。
- 每张卡片高度尽量紧凑。
- 时间显示为“今天 09:30”“昨天 21:10”。
- 内容超过两行省略，详情页完整展示。

## 7. 教师端微信小程序页面设计

### 7.1 教师审核列表页

目标：教师能像处理待办一样快速审核。

布局：

```text
顶部：班级审核

统计条：
- 待审核 12
- 已通过 30
- 已拒绝 2

筛选：
[待审核] [已通过] [已拒绝] [全部]

学生打卡卡片：
- 学生姓名 + 班级
- 打卡类型 + 状态
- 提交时间
- 内容摘要
- 操作：查看
```

设计要点：

- 默认只显示“待审核”。
- 卡片右上角显示状态 Chip。
- 教师列表页只做快速扫读，具体操作放到详情页。

### 7.2 教师审核详情页

目标：教师看清学生提交内容，并完成通过/拒绝。

布局：

```text
学生信息：
- 姓名
- 班级
- 打卡类型
- 提交时间

打卡内容：
- 文字心得
- 照片
- 定位信息

审核意见：
- 评语输入框

底部操作：
[拒绝] [通过]
```

交互：

- 点击“通过”：如果没有评语，也允许通过。
- 点击“拒绝”：必须填写评语。
- 操作成功后返回审核列表，并从待审核列表中移除该记录。

### 7.3 班级概览页

目标：教师快速知道今天班级整体情况。

布局：

```text
顶部：班级概览

数据卡：
- 今日打卡率
- 未打卡人数
- 待审核人数

未打卡名单：
- 姓名
- 班级
- 最近一次打卡时间
```

MVP 可简化：

- 如果时间不足，班级概览可以暂时并入审核列表顶部统计条。

| 系统日志页 | V1.1 或内部接口 | 胡钊炫 |

### 11.2 模块隔离

- 学生小程序只调用学生相关接口，不直接出现教师审核入口。
- 教师小程序只处理班级审核和概览，不承载管理端功能。
- 管理端 Web 不做学生提交打卡，只做查看、统计、管理。
- 三端共用同一套后端 API 鉴权逻辑。
- 页面组件只消费接口返回数据，不直接拼接后端业务规则。

## 13. 验收标准

### 13.1 学生端

- 学生能在 3 次点击内进入提交打卡页。
- 提交按钮在缺少必填信息时不可提交或明确提示。
- 打卡提交后能在历史页看到状态。
- 待审核、已通过、已拒绝状态颜色清晰。

### 13.2 教师端

- 教师打开首页即可看到待审核数量。
- 教师能进入详情页查看文字、照片、定位。
- 拒绝时必须填写评语。
- 审核完成后列表状态及时更新。

### 13.3 管理端

- 管理员登录后默认进入数据看板。
- 今日打卡率、未打卡人数、待审核数量可见。
- 表格筛选条件清晰，分页可用。
- 系统日志页面能按级别和 request_id 查询，MVP 可延后到 P1。

## 14. 不做范围

以下内容不进入 72 小时 MVP UI：

- AI 内容审核配置页面。
- 人脸识别页面。
- 积分排行榜。
- 复杂组织架构管理。
- 大屏展示模式。
- 多主题切换。
- 高级报表导出配置。

这些功能进入 V1.1 或后续版本。
