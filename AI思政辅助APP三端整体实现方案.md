# AI思政辅助APP三端整体实现方案

## 一、总体判断

这套系统不应按“查寝、课堂、实习”三个固定功能去开发，而应按“可配置打卡任务平台”来实现。

核心抽象是：

> 管理员配置组织、角色、打卡类型、规则模板和系统策略；教师基于班级或分组创建任务；学生按任务规则完成打卡；系统沉淀记录、异常、申诉、统计和风险提示。

因此第一阶段的实现重点不是做很多页面，而是先把以下能力打牢：

1. 组织与用户体系
2. 班级与分组体系
3. 打卡类型配置
4. 规则模板配置
5. 打卡任务创建与执行
6. 打卡记录、异常、申诉闭环
7. 基础权限与数据范围控制

只要这些底层模型设计好，后续新增“早操打卡、返校确认、安全上报、活动签到”等场景，主要是新增打卡类型和规则模板，而不是重新开发一套业务。

## 二、端侧定位与技术形态

## 1. 学生端

建议形态：微信小程序或移动端 H5。

核心任务：

- 查看今日任务
- 完成打卡
- 接收提醒和反馈
- 提交异常说明或申诉
- 查看个人记录和成长档案

一级导航建议保持为：

> 首页｜打卡任务｜消息｜我的

学生端重点是“轻、快、明确”。不要让学生理解复杂规则，只需要告诉他当前任务需要做什么、为什么失败、下一步怎么处理。

## 2. 教师端

建议形态：微信小程序，也可后期补充 Web 管理页。

核心任务：

- 创建打卡任务
- 使用模板创建任务
- 查看班级完成情况
- 处理异常和申诉
- 管理班级学生数据

一级导航建议为：

> 首页｜任务｜班级｜我的

教师端重点是“创建任务”和“处理异常”。创建任务要通过模板和分步表单降低配置成本。

## 3. 管理员端

建议形态：Web 后台。

核心任务：

- 维护组织、用户、班级、分组
- 配置打卡类型和规则模板
- 管理角色权限
- 监管任务、异常和申诉
- 查看统计分析和操作日志

管理员端重点是“平台治理”，不是教师端的放大版。

## 三、推荐系统架构

## 1. 前端

推荐拆成两个前端项目：

1. `admin-web`
   - 管理员 Web 后台
   - 可使用 Vue3 + Element Plus，或 React + Ant Design
   - 适合表格、筛选、权限、配置、报表类页面

2. `mini-app`
   - 学生端和教师端小程序
   - 可使用原生微信小程序、uni-app、Taro
   - 学生与教师通过角色区分首页和功能入口

如果开发人力有限，建议：

- 管理员端先用 Vue3 + Element Plus
- 小程序端用 uni-app，一套代码承载学生端和教师端

## 2. 后端

推荐采用单体模块化后端，第一阶段不要一上来拆微服务。

可选技术：

- Java Spring Boot + MyBatis Plus
- Node.js NestJS + Prisma
- Python FastAPI + SQLAlchemy

如果是学校项目或毕业设计，推荐 Java Spring Boot，资料多、后台管理系统生态成熟、论文表达也更稳。

后端按模块拆分：

- 认证与权限模块
- 组织用户模块
- 班级分组模块
- 打卡类型模块
- 规则模板模块
- 任务模块
- 打卡记录模块
- 异常申诉模块
- 消息通知模块
- 数据统计模块
- 系统日志模块
- AI辅助规则模块

## 3. 数据库与缓存

第一阶段推荐：

- MySQL：核心业务数据
- Redis：验证码、动态码、短期任务状态、消息未读数
- 对象存储或本地文件服务：图片、人脸照、附件、日志图片

后期可增强：

- Elasticsearch：日志和复杂搜索
- ClickHouse：大规模统计分析
- 消息队列：定时提醒、统计异步计算

## 四、核心业务模型

## 1. 用户与角色

建议统一用户表，不要学生、教师、管理员完全拆散登录体系。

核心表：

- `user`
- `student_profile`
- `teacher_profile`
- `admin_profile`
- `role`
- `permission`
- `user_role`

用户基础字段：

- 用户ID
- 账号
- 手机号
- 密码或 openid
- 用户类型：student、teacher、admin
- 状态：启用、禁用、锁定
- 最近登录时间

学生扩展字段：

- 姓名
- 学号
- 学院
- 专业
- 年级
- 行政班
- 宿舍
- 实习单位
- 当前状态：在校、实习、请假、离校

教师扩展字段：

- 姓名
- 工号
- 院系
- 教师角色
- 管理班级范围

管理员扩展字段：

- 所属部门
- 管理范围
- 数据权限级别

## 2. 组织结构

用树结构维护学校、学院、专业、年级、班级。

核心表：

- `organization`

关键字段：

- 组织ID
- 组织名称
- 组织类型：school、department、college、major、grade、class
- 父级组织ID
- 负责人ID
- 排序
- 状态

## 3. 班级与分组

班级和分组建议抽象成统一的 `group`。

核心表：

- `group`
- `group_member`
- `group_teacher`

分组类型：

- 行政班
- 课程班
- 实习小组
- 活动分组
- 自定义分组

这样教师创建任务时，只需要选择一个或多个 group，不需要关心它到底是行政班还是课程班。

## 4. 打卡类型

打卡类型由管理员维护，是任务的业务场景分类。

核心表：

- `checkin_type`

关键字段：

- 类型ID
- 类型名称
- 类型编码
- 适用场景
- 图标
- 默认说明
- 是否系统预设
- 是否启用
- 允许教师自定义
- 可用角色
- 适用组织范围
- 默认模板ID

典型数据：

- 查寝类：`dorm_check`
- 课堂类：`class_check`
- 实习类：`internship_check`
- 活动类：`activity_check`
- 会议类：`meeting_check`
- 早操类：`morning_check`
- 返校类：`return_check`
- 安全上报类：`safety_report`
- 自定义类：`custom_check`

## 5. 规则模板

规则模板建议采用“结构化字段 + JSON 配置”的混合方式。

核心表：

- `rule_template`
- `rule_template_version`

关键字段：

- 模板ID
- 模板名称
- 模板类型：系统、学院、教师个人
- 适用打卡类型
- 适用组织
- 创建人
- 状态
- 使用次数
- 规则 JSON

规则 JSON 示例：

```json
{
  "timeRule": {
    "mode": "single",
    "startTime": "21:30",
    "endTime": "22:30",
    "allowLate": false,
    "allowMakeup": true,
    "makeupNeedReview": true
  },
  "locationRule": {
    "mode": "fixed_area",
    "placeName": "3号宿舍楼",
    "longitude": 120.000001,
    "latitude": 30.000001,
    "radius": 300,
    "allowExceptionSubmit": true
  },
  "verificationRule": {
    "methods": ["location", "face"]
  },
  "submitRule": {
    "fields": [
      {
        "key": "remark",
        "label": "情况说明",
        "type": "textarea",
        "required": false
      }
    ]
  },
  "reviewRule": {
    "mode": "exception_only"
  },
  "reminderRule": {
    "beforeStartMinutes": 10,
    "beforeEndMinutes": 5,
    "notifyTeacherAfterEnd": true
  }
}
```

## 6. 打卡任务

任务是教师发布给班级或分组的具体业务对象。

核心表：

- `checkin_task`
- `checkin_task_target_group`
- `checkin_task_target_student`

任务关键字段：

- 任务ID
- 任务名称
- 任务说明
- 打卡类型ID
- 发布教师ID
- 所属组织ID
- 规则快照 JSON
- 任务状态：草稿、未开始、进行中、已结束、已归档、已强制结束
- 发布时间
- 开始时间
- 结束时间
- 是否周期任务
- 是否发送通知
- 是否保存为模板

注意：任务创建时必须保存一份规则快照，不要只引用模板ID。因为模板后续可能被管理员修改，历史任务应保持当时的规则不变。

## 7. 打卡记录

学生每次提交生成打卡记录。

核心表：

- `checkin_record`
- `checkin_record_attachment`
- `checkin_record_form_value`

关键字段：

- 记录ID
- 任务ID
- 学生ID
- 提交时间
- 打卡状态：正常、迟到、未打卡、异常、待审核、已通过、已驳回
- 定位经纬度
- 定位校验结果
- 人脸识别结果
- 动态码校验结果
- 提交内容 JSON
- 系统判定结果
- 最终审核结果

## 8. 异常与申诉

异常和申诉要形成完整闭环。

核心表：

- `checkin_exception`
- `appeal`
- `review_log`

异常类型：

- 未打卡
- 超时打卡
- 定位异常
- 人脸识别失败
- 动态码错误
- 疑似代签
- 日志未提交
- 安全风险上报
- 教师超时未处理

处理流程：

1. 系统生成异常
2. 学生可提交说明或申诉
3. 教师审核处理
4. 管理员可监管和督促
5. 高级管理员可平台级复核

## 五、关键业务流程

## 1. 教师创建任务流程

1. 选择创建方式
   - 空白创建
   - 使用系统模板
   - 使用学院模板
   - 使用我的模板

2. 填写基础信息
   - 任务名称
   - 任务说明
   - 是否通知学生
   - 是否保存为模板

3. 选择发布对象
   - 行政班
   - 课程班
   - 实习小组
   - 活动分组
   - 自定义学生

4. 选择打卡类型
   - 查寝类
   - 课堂类
   - 实习类
   - 活动类
   - 自定义类

5. 配置任务规则
   - 时间规则
   - 地点规则
   - 身份校验
   - 提交内容
   - 审核规则
   - 提醒规则

6. 确认发布
   - 保存草稿
   - 定时发布
   - 立即发布

## 2. 学生打卡流程

1. 学生首页看到待完成任务
2. 进入任务详情
3. 系统根据任务规则动态加载组件
   - 定位组件
   - 人脸组件
   - 动态码组件
   - 图片上传
   - 实习日志
   - 安全状态表单
4. 学生提交
5. 后端执行规则校验
6. 生成打卡记录
7. 正常记录自动通过
8. 异常记录进入待审核或申诉流程

## 3. 异常处理流程

1. 系统识别异常
2. 教师首页和异常页出现待处理事项
3. 教师查看学生打卡详情
4. 教师选择处理结果
   - 审核通过
   - 审核驳回
   - 要求补充说明
   - 标记高风险
5. 学生收到处理结果
6. 管理员端同步看到处理状态和处理时长

## 六、权限设计

权限不能只控制菜单，还要控制数据范围。

## 1. 权限维度

建议拆成五类：

1. 菜单权限
   - 能否进入某个页面

2. 操作权限
   - 能否新增、编辑、删除、审核、导出

3. 数据范围权限
   - 全校
   - 本学院
   - 本专业
   - 本班级
   - 本人创建

4. 敏感信息权限
   - 定位信息
   - 人脸结果
   - 手机号
   - 申诉详情

5. 导出权限
   - 是否允许导出
   - 是否需要审批
   - 导出字段是否脱敏

## 2. 推荐角色

管理员端：

- 系统超级管理员
- 学校管理员
- 学院管理员
- 学工管理员
- 教务管理员
- 实习管理员
- 数据审计员

教师端：

- 辅导员
- 班主任
- 任课教师
- 实习指导教师
- 学院管理员教师角色

学生端：

- 普通学生
- 学生干部，可选

## 八、规则引擎实现方案

第一阶段不建议引入复杂规则引擎，可以使用“规则配置 JSON + 后端策略类”的方式。

后端处理步骤：

1. 读取任务规则快照
2. 校验时间规则
3. 校验地点规则
4. 校验身份规则
5. 校验提交内容
6. 生成系统判定结果
7. 根据审核规则决定是否自动通过
8. 如异常则生成异常记录

示例策略：

- `TimeRuleEvaluator`
- `LocationRuleEvaluator`
- `VerificationRuleEvaluator`
- `SubmitRuleEvaluator`
- `ReviewRuleEvaluator`

返回统一结果：

```json
{
  "passed": false,
  "status": "exception",
  "exceptionTypes": ["location_error"],
  "messages": ["当前位置不在有效范围内"],
  "needReview": true
}
```

这样后续新增规则时，只需要新增 evaluator，而不破坏任务主流程。

## 九、AI能力实现边界

AI能力第一阶段不要做成“自动处分”或“自动定责”，而是做辅助判断。

建议先实现这些轻量能力：

1. 定位异常辅助判断
   - 超出范围
   - 定位漂移
   - 多次异常

2. 疑似代签规则判断
   - 同设备多人打卡
   - 短时间同位置多人异常
   - 动态码多次错误

3. 实习安全风险关键词
   - 从实习日志或安全上报中识别风险词
   - 只生成提醒，不直接判定学生违规

4. 风险预警
   - 班级异常率高于近期平均
   - 教师长时间未处理异常
   - 学生连续未打卡

必须在产品上明确：

- AI只做辅助判断
- 最终处理由教师或管理员确认
- 学生必须有申诉通道
- 定位、人脸、日志等敏感数据要有授权说明

## 十三、实现风险与建议

## 1. 最大风险：规则配置过度复杂

建议第一版只支持常用规则组合：

- 单次时间段
- 固定地点半径
- 定位、动态码、图片、文字说明
- 正常自动通过，异常人工审核

周期任务、多时间段、复杂自定义表单可以第二阶段再做。

## 2. 最大技术点：任务规则快照

任务必须保存发布时的规则快照。否则模板一改，历史任务就会被影响，统计和申诉都会混乱。

## 3. 最大产品点：教师创建任务要简单

教师端不要一开始展示所有规则。建议用模板先填默认值，再允许展开高级配置。

## 4. 最大合规点：敏感数据

定位、人脸、手机号、申诉说明都属于敏感信息。需要：

- 授权说明
- 数据脱敏
- 查看权限
- 导出日志
- 操作留痕

## 5. 最大扩展点：打卡类型和规则模板

打卡类型、规则模板和任务规则快照是系统可扩展性的核心。后续能不能低成本新增场景，主要取决于这里设计得是否清楚。

## 十四、建议的毕业设计/项目表述

如果这个项目用于论文或答辩，可以把系统核心创新点表达为：

1. 面向多场景思政管理的可配置打卡任务模型
2. 基于角色与组织范围的数据权限控制
3. 支持班级、课程班、实习小组等多分组对象的任务发布机制
4. 基于规则模板的任务快速创建机制
5. 面向异常打卡和学生申诉的闭环处理流程
6. AI辅助风险预警与人工复核结合的学生管理方式

最终系统定位可以写成：

> 本系统构建了一个面向高校思政与学生日常管理的多端协同平台，通过管理员端完成组织、用户、权限、打卡类型和规则模板配置，通过教师端实现班级任务发布、数据查看和异常处理，通过学生端完成打卡、消息接收、申诉和个人档案查看，从而形成“规则配置—任务发布—学生执行—异常处理—数据分析”的闭环管理流程。
