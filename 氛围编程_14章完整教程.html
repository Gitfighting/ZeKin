第1章 什么是氛围编程

Vibe Coding — 2025年最火的编程新范式

1.1 Vibe Coding的定义

氛围编程（Vibe Coding）是2025年2月由OpenAI联合创始人Andrej Karpathy提出的新概念。核心理念是：程序员不用手写每一行代码，而是用自然语言描述需求，让AI来生成代码。

💡 简单理解：传统编程就像自己做菜——洗菜、切菜、炒菜全靠自己。氛围编程就像请了一位米其林大厨住在家里，你只需要说"今晚想吃红烧肉、少放盐、多放糖"，大厨就帮你做好了。你不需要会颠勺，但你需要知道自己想吃什么，以及菜端上来能不能吃。

1.2 传统编程 vs 氛围编程

对比维度 传统编程 氛围编程
流程 理解原理 → 写代码 → 调试 → 测试 描述需求 → AI生成 → 审核修改 → 快速出活
速度 慢，但基础扎实 快，但需要理解能力
核心能力 记住语法、理解算法、手写代码 清晰表达需求、判断代码质量、快速迭代
适合场景 深入学习编程原理、面试准备 快速做出产品原型、创业项目、学习项目
类比 自己种地、自己做饭——全靠自己动手 当产品经理，指挥AI干活——你负责想，它负责做
⚠️ 重要提醒：氛围编程不等于"不会编程"！恰恰相反，氛围编程需要你有更好的系统思维和判断力。就像一个优秀的将军不需要亲自拿枪射击，但他必须懂得战略、能判断战场形势。

1.3 AI编程工具介绍

  • Claude Code — Anthropic公司推出的AI编程助手，装在终端（命令行）里，可以直接读取项目文件、帮你写代码、修改文件、运行命令。
  • Codex — OpenAI（ChatGPT的公司）推出的AI编程工具，思路和Claude Code类似。

1.4 氛围编程的三个核心原则

原则一：你是产品经理，AI是程序员

你的角色是产品经理——你负责"想"，AI负责"做"。产品经理不需要会写代码，但必须：

  • 想清楚要做什么：需求要明确，不能说"帮我做个好看的网站"
  • 能判断做得对不对：AI生成代码后，要能看懂大概逻辑
  • 及时反馈和调整：发现问题就说出来，让AI修改

原则二：提示词是代码

在传统编程中，你写Python代码告诉计算机做什么。在氛围编程中，你写提示词（Prompt）告诉AI做什么。提示词写得好，AI输出的代码质量就高。

原则三：理解比记忆重要

传统编程需要记住大量语法。氛围编程中，AI帮你处理这些细节。你需要做的是：

  • 理解概念：知道什么是数据库、什么是API、什么是前端后端
  • 理解架构：知道一个项目由哪些部分组成，各部分怎么配合
  • 理解原理：知道为什么用这个方案而不是那个

1.5 最佳实践与常见误区

最佳实践 常见误区
小步迭代：每次只做一个改动，验证通过后再做下一个 以为AI生成的代码不需要检查：AI可能会"发明"不存在的依赖包
保留"决策日志"：记录每次让AI做了什么、为什么这样做 忽略安全：AI可能会硬编码API密钥、把密码明文存在数据库里
知道什么时候该停：氛围编程擅长从0到1，但不太擅长最后10%的打磨 "Vibe Coding宿醉"：快速做出能用的东西，第二天发现代码结构一团糟

3.5 与AI对话的技巧

差的提示词 好的提示词
"帮我做个登录功能" "帮我用FastAPI创建一个用户登录API接口，接收手机号和密码，验证后返回JWT Token。密码用bcrypt验证，Token过期时间24小时。"

⚡ 关键技巧：不要一次让AI做太多事。把大任务拆成小步骤，一步步来。每完成一步，确认无误后再进行下一步。

第4章 需求分析与提示词工程

把你的想法变成AI能理解的指令

4.1 提示词五要素

要素 说明 示例
1. 角色（Role） 告诉AI它是谁 "你是一个资深的Python后端工程师"
2. 任务（Task） 明确要做什么 "帮我创建一个FastAPI用户注册接口"
3. 背景（Context） 提供项目背景信息 "这是一个高校思政打卡小程序..."
4. 约束（Constraints） 说明限制和要求 "密码用bcrypt哈希，API用RESTful风格"
5. 输出格式（Format） 定义答案形式 "请输出完整的代码文件，包含注释说明"

4.2 MoSCoW优先级方法

MoSCoW是一种需求优先级排序方法：

  • M - Must（必须有）：没有就不能用的功能
  • S - Should（应该有）：没有会很难用的功能
  • C - Could（可以有）：有了更好的功能
  • W - Won't（这次不做）：留给以后的功能

第5章 数据库设计

盖房子先打地基，写代码先设计数据库

5.1 为什么要先设计数据库？

数据库就是程序的地基。如果地基没打好，后面写的所有代码都可能需要推翻重来。

思政打卡系统需要存储的数据：

  • 用户信息（学生、老师的账号）
  • 打卡记录（谁在什么时候打了什么卡）
  • 人脸特征（用来验证身份）
  • 班级信息（学生属于哪个班）
  • 审核记录（老师审核学生打卡）

5.2 核心表设计

1. users 用户表

字段名 类型 说明
id INT (主键) 用户的唯一编号，自动递增
username VARCHAR(50) 登录用户名，不能重复
email VARCHAR(100) 邮箱地址
hashed_password VARCHAR(255) 加密后的密码（绝对不存明文！）
real_name VARCHAR(50) 真实姓名
class_id INT (外键) 所属班级的编号
is_active BOOLEAN 账号是否启用
is_admin BOOLEAN 是否是管理员
created_at DATETIME 注册时间
⚠️ 安全提醒：密码字段叫 hashed_password，不是 password！我们绝不存储用户的原始密码。密码会经过 bcrypt 加密后存储。

2. checkins 打卡记录表

字段名 类型 说明
id INT (主键) 打卡记录的唯一编号
user_id INT (外键) 谁打的卡 → 关联 users 表
type ENUM 打卡类型：查寝/上课/实习
content TEXT 打卡内容描述
photo_url VARCHAR(255) 照片存储地址
latitude FLOAT 纬度（GPS定位）
longitude FLOAT 经度（GPS定位）
status ENUM 状态：待审核/已通过/已拒绝
created_at DATETIME 打卡时间

3. face_encodings 人脸特征表

字段名 类型 说明
id INT (主键) 特征记录的唯一编号
user_id INT (外键) 谁的人脸 → 关联 users 表
encoding BLOB 128 维特征向量
created_at DATETIME 录入时间
💡 128维特征向量：把一张人脸照片转换成128个数字。你可以把它想象成用128个特征来"描述"这张脸：眼睛的大小、鼻子的形状、嘴巴的宽度…… 这128个数字组合在一起，就能唯一地代表一张脸。

4. classes 班级表

字段名 类型 说明
id INT (主键) 班级编号
name VARCHAR(50) 班级名称，如"计科2301班"
teacher_id INT (外键) 班主任 → 关联 users 表

5. reviews 审核表

字段名 类型 说明
id INT (主键) 审核记录编号
checkin_id INT (外键) 哪条打卡 → 关联 checkins 表
reviewer_id INT (外键) 谁审核的 → 关联 users 表
status ENUM 审核结果：通过/拒绝
comment TEXT 审核意见
created_at DATETIME 审核时间

6. allowed_locations 允许打卡位置表

字段名 类型 说明
id INT (主键) 位置编号
name VARCHAR(100) 位置名称，如"3号宿舍楼"
latitude FLOAT 中心点纬度
longitude FLOAT 中心点经度
radius INT 允许范围（米）

5.3 ER关系图

表与表之间通过外键建立联系：

  • users → checkins（一对多）：一个用户可以有多条打卡记录
  • users → face_encodings（一对多）：一个用户可以录入多张人脸照片
  • classes → users（一对多）：一个班级有多个学生
  • checkins → reviews（一对多）：一条打卡可以被多次审核 
classes ──1:N──▶ users ──1:N──▶ checkins ──1:N──▶ reviews
              │
              └──1:N──▶ face_encodings

5.5 数据库设计最佳实践

  • 索引设计：经常查询的字段加索引，不该加的不加
  • 命名规范：表名复数、列名snake_case、外键以_id结尾

  • 软删除：用is_deleted字段标记，不要真的DELETE

6.4 API路由（Routers）

from fastapi import APIRouter, Depends, HTTPException
from app.schemas.user_schema import UserCreate, UserLogin, Token
from app.services.auth_service import register_user, login_user

router = APIRouter(prefix="/api/auth", tags=["认证"])

@router.post("/register", response_model=Token)
async def register(user_data: UserCreate):
    result, error = await register_user(user_data)
    if error:
        raise HTTPException(status_code=400, detail=error)
    return result

@router.post("/login", response_model=Token)
async def login(user_data: UserLogin):
    result, error = await login_user(user_data)
    if error:
        raise HTTPException(status_code=401, detail=error)
    return result

6.5 认证系统

JWT（JSON Web Token） — 身份认证方式，用户登录后获得"电子身份证"。

密码加密 — bcrypt

import bcrypt

# 注册时：加密密码
password = "mypassword123"
hashed = bcrypt.hashpw(password.encode(), bcrypt.gensalt())

# 登录时：验证密码
is_correct = bcrypt.checkpw(password.encode(), hashed)

Token管理

import hashlib, secrets

async def create_token(user_id: int, redis) -> str:
    token = secrets.token_hex(32)
    token_hash = hashlib.sha256(token.encode()).hexdigest()
    await redis.set(f"token:{token_hash}", str(user_id), ex=86400)
    return token

async def verify_token(token: str, redis) -> int:
    token_hash = hashlib.sha256(token.encode()).hexdigest()
    user_id = await redis.get(f"token:{token_hash}")
    if not user_id:
        raise HTTPException(status_code=401, detail="Token无效")
    return int(user_id)

        raise HTTPException(status_code=400, detail="只能上传图片文件")
    file_ext = os.path.splitext(file.filename)[1]
    filename = f"{current_user.id}_{int(time.time())}{file_ext}"
    filepath = f"uploads/{filename}"
    with open(filepath, "wb") as buffer:
        shutil.copyfileobj(file.file, buffer)
    return {"url": f"/uploads/{filename}"}

7.5 人脸比对验证

async def verify_face(user_id: int, photo_path: str) -> dict:
    # 提取打卡照片的人脸特征
    image = face_recognition.load_image_file(photo_path)
    face_locations = face_recognition.face_locations(image)
    if len(face_locations) == 0:
        return None, "未检测到人脸"

    unknown_encodings = face_recognition.face_encodings(image, face_locations)
    unknown_encoding = unknown_encodings[0]

    # 与数据库中存储的特征比对
    known_encodings = await get_user_face_encodings(user_id)

    for known_encoding in known_encodings:
        matches = face_recognition.compare_faces(
            [known_encoding], unknown_encoding, tolerance=0.6
        )
        if matches[0]:
            return {"verified": True, "message": "验证通过"}, None

    return None, "人脸验证失败"

// 封装API请求
const API_BASE = '/api'

async function request(url, options = {}) {
  const token = localStorage.getItem('token')
  const headers = {
    'Content-Type': 'application/json',
    ...(token ? { 'Authorization': `Bearer ${token}` } : {})
  }
  const response = await fetch(`${API_BASE}${url}`, {
    ...options,
    headers: { ...headers, ...options.headers }
  })
  if (response.status === 401) {
    localStorage.removeItem('token')
    window.location.href = '/login'
    return
  }
  return response.json()
}

10.1 打卡完整流程（6步）

  1. 获取GPS位置
  2. 判断地理围栏
  3. 打开摄像头拍照
  4. 上传照片+位置

  6. 返回打卡结果

10.2 地理围栏（Haversine公式）

计算地球表面两点之间的距离，判断是否在围栏内。

import math

def haversine(lat1, lon1, lat2, lon2):
    """计算两个GPS坐标之间的距离（米）"""
    R = 6371000  # 地球平均半径（米）
    lat1_rad = math.radians(lat1)
    lat2_rad = math.radians(lat2)
    dlat = math.radians(lat2 - lat1)
    dlon = math.radians(lon2 - lon1)
    a = (math.sin(dlat / 2) ** 2 +
         math.cos(lat1_rad) * math.cos(lat2_rad) *
         math.sin(dlon / 2) ** 2)
    c = 2 * math.asin(math.sqrt(a))
    return R * c

def is_in_geofence(user_lat, user_lng, target_lat, target_lng, radius):
    distance = haversine(user_lat, user_lng, target_lat, target_lng)
    return distance <= radius, distance

10.3 不同场景的围栏半径

场景 推荐半径 说明
教室打卡 50米 教室面积有限
宿舍打卡 100米 宿舍楼面积较大
操场打卡 200米 操场面积大
实习地点 150米 企业园区范围
⚠️ 现实考虑：GPS定位本身有误差（室外5-15米，室内30-100米），围栏半径不能设太小。宿舍楼内GPS精度差，建议用Wi-Fi指纹或蓝牙辅助定位。

⚡ 审核加速技巧：① 提供可用的测试账号密码 ② 备注中写清楚功能流程和测试步骤 ③ 工作日上午提交（审核队列较短）

12.5 正式发布

  • 全量发布：所有用户立即更新
  • 灰度发布：先让一部分用户体验，没问题再全量

版本号遵循语义化版本号：MAJOR.MINOR.PATCH（主版本.次版本.补丁号）

第13章 AI辅助测试

用AI自动生成测试，保证代码质量

13.1 为什么要写测试？

  • 防止改A坏B：修改代码后跑测试，知道有没有影响其他功能
  • 自动验证：不用每次手动点来点去
  • 活文档：测试代码本身就是最好的使用说明
  • 放心重构：有测试兜底才敢优化代码

13.2 单元测试（pytest）

# tests/test_auth.py
import pytest
from httpx import AsyncClient, ASGITransport
from app.main import app

@pytest.fixture
async def client():
    transport = ASGITransport(app=app)
    async with AsyncClient(transport=transport, base_url="http://test") as c:
        yield c

@pytest.mark.asyncio
async def test_register_success(client):
    """测试：正常注册新用户"""
    response = await client.post("/api/auth/register", json={
        "username": "testuser",
        "password": "Test123456",
        "email": "test@example.com"
    })
    assert response.status_code == 200

@pytest.mark.asyncio
async def test_login_wrong_password(client):
    """测试：密码错误"""
    response = await client.post("/api/auth/login", json={
        "username": "testuser",
        "password": "WrongPassword"
    })
    assert response.status_code == 401

13.3 测试覆盖率

测试覆盖率 — 代码被测试到的比例。用pytest-cov分析。

# 安装
pip install pytest-cov

# 运行测试并生成覆盖率报告
pytest --cov=app tests/ -v

# 生成HTML报告（在浏览器中查看）
pytest --cov=app --cov-report=html tests/

13.4 推荐覆盖率目标

模块类型 推荐覆盖率 重点测试内容
API路由层（routers/） ≥ 90% 所有接口的正常和异常流程
业务逻辑层（services/） ≥ 85% 核心业务规则和边界条件
数据模型层（models/） ≥ 70% 字段验证、关系约束
工具函数（utils/） ≥ 95% 各种输入的返回值

13.5 AAA测试模式

Arrange（准备数据）→ Act（执行操作）→ Assert（断言结果）

14.3 Redis缓存

Redis三大用途：

  1. Session缓存：用户登录Token存在Redis，不用每次查数据库

14.4 CI/CD流水线

CI/CD — 代码提交后自动测试、自动部署的流程。

# .github/workflows/deploy.yml
name: CI/CD Pipeline

on:
  push:
    branches: [main]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: '3.11'
      - run: pip install -r requirements.txt
      - run: pytest tests/ -v --cov=app

  deploy:
    needs: test
    runs-on: ubuntu-latest
    steps:
      - name: Deploy via SSH
        uses: appleboy/ssh-action@v1
        with:
          host: ${{ secrets.SERVER_HOST }}
          username: ${{ secrets.SERVER_USER }}
          key: ${{ secrets.SSH_PRIVATE_KEY }}
          script: |
            cd /opt/checkin-app
            git pull origin main
            docker-compose up -d --build

14.5 Nginx反向代理

Nginx站在前台接客，收到请求后转交给后台的FastAPI处理。

# /etc/nginx/sites-available/checkin-app
server {
    listen 80;
    server_name your-domain.com;

    location /static/ {
        alias /opt/checkin-app/static/;
        expires 30d;
    }

    location /api/ {
        proxy_pass http://127.0.0.1:9000;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
    }
}

14.7 日志监控（loguru）

日志级别：DEBUG → INFO → WARNING → ERROR → CRITICAL

from loguru import logger
import sys

logger.remove()

# 控制台输出（带颜色）
logger.add(sys.stderr, level="DEBUG", format="...")

# 文件输出（每天一个文件，保留30天，自动压缩）
logger.add(
    "logs/app_{time:YYYY-MM-DD}.log",
    rotation="00:00",
    retention="30 days",
    compression="zip",
    level="INFO",
    encoding="utf-8"
)

# 错误日志单独存放
logger.add("logs/error_{time:YYYY-MM-DD}.log", level="ERROR")
